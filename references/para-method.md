# PARA 方法论

PARA 是一种通用的事务组织方法，适用于任何软件环境中的任何类型的个人或专业事务。

## 四大分类

### 1. Projects (项目)
- **定义**: 有明确目标和截止日期的任务
- **特点**: 需要多步骤完成，有明确的"完成"状态
- **示例**: "完成网站重构"、"学习 Python 基础"

### 2. Areas (领域)
- **定义**: 持续关注的责任范围
- **特点**: 没有截止日期，需要持续维护
- **示例**: "健康"、"财务"、"职业技能"

### 3. Resources (资源)
- **定义**: 有价值的参考资料和知识
- **特点**: 可复用，为未来参考而保存
- **示例**: "技术文章"、"教程"、"灵感素材"

### 4. Archives (归档)
- **定义**: 已完成或不再活跃的事务
- **特点**: 保留以备将来参考
- **示例**: "已完成项目"、"旧笔记"

## 文件夹映射

| PARA 分类 | Obsidian 目录 |
|----------|--------------|
| Projects | `10_项目/` |
| Areas | `20_领域/` |
| Resources | `30_研究/` + `40_知识库/` |
| Archives | `50_归档/` |

## 工作流

1. **收集**: 所有新内容进入 `00_收件箱/`
2. **处理**: 定期处理收件箱，分配到 PARA 分类
3. **回顾**: 定期回顾项目和领域状态
4. **归档**: 完成的项目移动到归档

## 核心原则

### 1. 按可执行程度组织
- Projects > Areas > Resources > Archives
- 越容易执行的内容越容易访问

### 2. 动态流动
- 项目完成后移动到归档
- 领域相关资源可随时转化为项目
- 定期回顾保持组织新鲜度

### 3. 上下文分离
- 不同项目可以共享同一领域
- 领域中的内容可服务于多个项目
- 归档不影响当前工作流程

## 实践建议

### 每周回顾
1. 检查收件箱
2. 更新项目状态
3. 审视领域关注点
4. 归档已完成事项

### 定期整理
- 清理过期笔记
- 合并重复内容
- 更新链接关系

### 知识积累
- 将项目经验沉淀为知识库条目
- 将研究笔记提炼为可复用知识
- 通过 MOC 建立知识网络
